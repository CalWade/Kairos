# Kairos 效能指标硬化测量

> 对齐赛题要求："量化展示成果（例如：使用前需要敲 50 个字符，使用后只需 10 个，提效 80%）"

## 场景定义

**场景**：团队 A/B 两人在飞书群里，A 问 B "咱们项目到底用 SQLite 还是 PostgreSQL？我记得上周讨论过。"

B 要回答 A 之前，需要找到历史决策。下面对比两种流程。

## 流程 A：无 Kairos 的手工流程

B 要做的事：

| # | 操作 | 耗时（中位数）| 输入字符 | 动作类别 |
|---|---|---:|---:|---|
| 1 | 切到飞书客户端前台 | 1s | 0 | 切换 |
| 2 | 找到目标项目群 / 或打开会话搜索 | 3s | 0 | 导航 |
| 3 | 点击搜索框 | 1s | 0 | 点击 |
| 4 | 输入搜索关键词 "PostgreSQL" 或 "SQLite" | 5s | **10 字符** | 输入 |
| 5 | 翻看搜索结果（≥5 条可能相关），识别出"最终决定"那条 | 10s | 0 | 翻阅 |
| 6 | 展开聊天上下文，确认决策理由（可能还要翻上下 10 条消息）| 15s | 0 | 翻阅 |
| 7 | 回到当前讨论，复制决策文本粘贴 | 4s | 0 | 粘贴 |

**累计**：
- **操作步数：7 步**
- **耗时：39 秒**（中位数估计）
- **键盘输入：10 个字符**（搜索关键词）
- **认知负担**：要记住是哪个群、关键词是什么、哪条才是"最终决定"

## 流程 B：有 Kairos 的流程

| # | 操作 | 耗时 | 输入字符 |
|---|---|---:|---:|
| 1 | Kairos 在 A 问出问题时就自动识别为复议，推送历史决策卡片 | **0s（系统自动）** | 0 |
| 2 | B 看到卡片，直接基于卡片里的决策 + 理由 + 被否方案回答 A | 3s | 0 |

**累计**：
- **操作步数：2 步**（只是"看+答"）
- **耗时：3 秒**
- **键盘输入：0 个字符**
- **认知负担**：零，卡片自动送到眼前

## 对比指标

| 指标 | 手工流程 | Kairos | 改善 |
|---|---:|---:|---:|
| 操作步数 | 7 | 2 | **-71.4%** |
| 耗时 | 39s | 3s | **-92.3%** |
| 键盘输入字符 | 10 | 0 | **-100%** |
| 认知负担 | 高（记群名/关键词/判断结果）| 零 | — |

## 额外维度：AI agent 场景

如果是 AI agent（如飞书内置助手、OpenClaw agent）代替 B 回答 A：

**手工流程对 agent 的负担**：agent 要调用 feishu search API、处理返回 JSON、排序、生成回答。约 **2-3 次 API 调用 + ~1500 tokens 消耗**。

**Kairos 流程**：agent 读取 MemoryAtom `subject + decision + reasons + rejected_options` 结构化字段，**1 次 Store.search 调用 + ~200 tokens**。

| 维度 | 手工（agent）| Kairos | 改善 |
|---|---:|---:|---:|
| API 调用次数 | 2-3 | 1 | -50%~-67% |
| Token 消耗 | ~1500 | ~200 | **-86.7%** |
| 结果可靠性 | 依赖 full-text search 排序 | 结构化字段直接取值 | 确定性 |

## 边界披露

- 流程 A 的秒数是"常见中位数估算"，不是严格用户研究数据。真实用户在不同场景下可能更快或更慢
- 流程 A 的 7 步是飞书标准搜索体验，有的群有 pin / 文档链接可以缩短到 4-5 步
- Kairos 依赖**历史记忆已经沉淀**，如果群里之前根本没讨论过这个主题，两种流程都无效
- Kairos 的"认知负担零"假设卡片弹出不过度打扰；实际 ActivationThrottle 15 分钟冷却控制频率

## 重现实验

想用自己的数据复现：

```bash
# 1. 往 Kairos 注入一条历史决策
npm run dev -- extract-decision --text "最终决定：复赛阶段使用 SQLite，不使用 PostgreSQL。理由：评委要能一键跑、部署轻。" --write

# 2. 模拟 B 重新提起复议
npm run dev -- eval --smoke   # smoke 包含这个场景

# 3. 看命中的历史决策卡片
# 输出中会有 activation.action = "push_decision_card" 和完整卡片 JSON
```
